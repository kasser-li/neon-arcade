## 1. 核心理念

### 1.1 为什么选择 SDD 而不是 Vibe Coding？

**Vibe Coding**（感觉驱动开发）：
- ✅ 开发速度极快
- ❌ 技术债务不可控
- ❌ 代码质量看运气
- ❌ 难以团队协作 Review
- ❌ 无文档沉淀

**SDD**（规格驱动开发）：
- ✅ 代码质量有保障
- ✅ 技术债务可控
- ✅ Spec 可评审、可追溯
- ✅ Spec 即文档，知识可沉淀
- ⚠️ 前期编写 Spec 有成本

**结论**：
> 企业级业务系统 → **必须 SDD**  
> 个人小玩具/原型验证 → Vibe Coding 随便玩

### 1.2 核心原则

```
📜 文档即真理（Spec is Truth）
🤖 代码只是文档的实现
✍️ 修改先改 Spec，再让 AI 同步代码
🚫 严禁手工修改代码（紧急情况除外）
🔄 Spec 和代码必须保持一致
```

## 2. 工具链选型

### 2.1 推荐组合（已验证）

| 类别 | 工具 | 版本 | 说明 |
|------|------|------|------|
| IDE | TRAE | 最新版 | 国内版 VS Code 定制 |
| 规范工具 | OpenSpec | 原版 | 不要 cn 版，不要 template |
| 测试工具 | Playwright | 最新版 | 网页自动化测试 |
| 测试报告 | Excel MCP | - | 生成 Excel 测试报告 |
| 代码生成 | OpenSpec + Rules | - | 自定义 Rules 目录 |

### 2.2 为什么不选其他工具？

| 工具 | 定位 | 问题 | 结论 |
|------|------|------|------|
| Spec-Kit | 理想主义规范工具 | 流程太严格，拖慢节奏 | ❌ |
| Superpowers | 执行层 AI 工作流 | 不沉淀需求规格 | ❌ |
| openspec-cn | 汉化版本 | 版本滞后，功能裁剪 | ❌ |
| fullstack-template | 脚手架模板 | 约束太强，限制架构 | ❌ |

### 2.3 开发环境要求

**后端**：
- JDK 21（新项目）/ JDK 8（旧项目）
- Maven 3.9.x+ / Gradle 9.3.x+
- MySQL 8.0+
- Redis 6.x+

**前端**：
- Node 24.x.x+
- Npm 11.x.x+ / Yarn 1.22.x+
- Vue 3.x（Vite + TypeScript）

**Python 环境**：
- Python 3.11.x+

### 2.4 TRAE 配置要点

**关键配置**：
```
设置项：java.autobuild.enabled
值：false
原因：避免与 Maven 打包冲突
```

**操作步骤**：
1. 点击 Trae 左下角 Java 图标
2. 找到 "Build Automatically"
3. 关闭它
4. 或在设置中搜索 `java.autobuild.enabled` 设为 false

**影响**：
- ✅ 解决 Maven 打包失败问题
- ⚠️ 修改代码后需要手动编译

## 3. 标准作业流程

### 3.1 项目初始化

```bash
# 步骤 1：项目根目录执行
openspec init

# 步骤 2：生成项目规范
# 在 Trae Builder 模式下对话：
"请读取项目代码，生成 openspec/config.yaml 和 rules/目录规范"
```

**产出物**：
```
.openspec/
├── config.yaml        # 项目配置
├── rules/             # 代码规范约束
│   ├── 00-general.md
│   ├── 10-backend.md
│   ├── 20-frontend.md
│   └── ...
└── changes/           # 变更管理
    ├── active/        # 进行中的变更
    └── archive/       # 已归档的变更
```

### 3.2 需求开发七步法

```
┌─────────────────────────────────────────────────────────┐
│  Step 1: explore（探索需求）                            │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  Step 2: proposal（生成提案）                           │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  Step 3: apply（生成代码）                              │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  Step 4: test（测试验证）                               │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  Step 5: sync（反向同步）                               │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  Step 6: verify（一致性检查）                           │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  Step 7: archive（归档）                                │
└─────────────────────────────────────────────────────────┘
```

### 3.3 各步骤详解

#### **Step 1: explore（探索需求）**

**命令**：
```bash
/opsx:explore
```

**提示词示例**：
```
我想实现一个客户管理功能，包括客户新增、编辑、删除、查询，
客户字段包括：姓名、电话、邮箱、地址、备注。
请帮我分析需求，考虑是否有遗漏的场景。
```

**目的**：
- 和 AI 聊需求，让它出谋划策
- 只读模式，不修改文件
- 理清需求细节和边界场景

---

#### **Step 2: proposal（生成提案）**

**命令**：
```bash
/opsx:propose
```

**产出物**：
```
.openspec/changes/active/xxx-feature/
├── proposal.md      # 需求提案
├── specs/           # 详细规格
│   ├── api.md       # 接口定义
│   └── data.md      # 数据模型
├── design.md        # 设计方案
└── tasks.md         # 任务分解
```

**⚠️ 关键**：**人工审核！**
- 检查需求理解是否准确
- 检查接口设计是否合理
- 检查数据模型是否完整
- 有问题继续对话让 AI 修改

---

#### **Step 3: apply（生成代码）**

**命令**：
```bash
/opsx:apply
```

**AI 自动完成**：
- 根据 Spec 生成 Controller、Service、Mapper
- 生成前端页面和 API 调用
- 生成数据库 SQL 脚本
- 生成单元测试代码

**⚠️ 关键**：**人工 Review！**
- 检查代码规范
- 检查代码风格
- 检查是否有安全隐患
- 发现问题让 AI 修改

---

#### **Step 4: test（测试验证）**

**测试清单**：
1. ✅ 编译打包，修复报错
2. ✅ 生成单元测试/集成测试
3. ✅ AI 自动执行测试并修复
4. ✅ 手动建表（SQL 已生成）
5. ✅ 手动启动前后端应用
6. ✅ Playwright 网页自动化测试

**Playwright 测试提示词**：
```
@网页测试专家
请对 [页面 URL] 进行自动化测试，覆盖以下场景：
1. 新增客户（正常场景）
2. 编辑客户（正常场景）
3. 删除客户（带确认弹窗）
4. 查询客户（模糊搜索）
5. 空表单提交（异常场景）
6. 重复提交（异常场景）

输出测试报告和 Excel 用例文件。
```

---

#### **Step 5: sync（反向同步）**

**⚠️ 重要！** 修改代码后必须反向更新 Spec

**提示词**：
```
在测试过程中发现以下问题：
1. 接口参数需要增加 xxx 字段
2. 数据库表需要增加 yyy 字段
3. 前端页面需要增加 zzz 功能

请同步更新 Spec 文件，保证 Spec 和代码一致。
```

**目的**：
- 保证 Spec 和代码一致
- 方便后续追溯和维护
- 新人可以通过 Spec 理解系统

---

#### **Step 6: verify（一致性检查）**

**命令**：
```bash
/opsx:verify
```

**目的**：
- 让 OpenSpec 从规范角度再次检查
- 确保没有遗漏
- 确保符合 Rules 约束

---

#### **Step 7: archive（归档）**

**命令**：
```bash
/opsx:archive
```

**作用**：
- 将 `changes/active/` 移动到 `changes/archive/`
- 按日期归档
- 保持目录整洁
- 保留完整历史记录

## 4. Rules 编写规范

### 4.1 核心原则

```
✅ 单个文件 ≤ 200 行（约 1000-1500 字符）
✅ 规则放在提示词开头和结尾（AI 最敏感）
✅ 通用规则放 Rules，复杂工作流放 Skills
✅ 按需加载，不要全部常驻
```

### 4.2 文件结构

```
.openspec/rules/
├── 00-general.md        # 通用规范（所有角色都遵守）
├── 10-backend.md        # 后端开发规范
├── 20-frontend.md       # 前端开发规范
├── 30-database.md       # 数据库规范
├── 40-testing.md        # 测试规范
└── 50-security.md       # 安全规范
```

### 4.3 编写技巧

**权重问题**：
> 大语言模型对提示词的**开头和结尾**最敏感，对中间内容关注度较低

**指令遵循度**：
> 规则文件太长 → AI 遵循度下降 → 可能只记住第一条和最后一条

**最佳实践**：
```markdown
# 后端开发规范

## 必须遵守（放开头）
- 使用 4 个空格缩进
- 命名使用驼峰式
- 禁止硬编码魔法值

## ... 中间内容 ...

## 重要提醒（放结尾）
- 所有接口必须返回统一响应结构
- 必须包含单元测试
- 必须处理异常
```

### 4.4 持续优化

**每次开发后**：
```
提示词：
"在这次开发中，发现 AI 生成了魔法值/命名不规范/缺少注释
请根据这些问题，优化项目的 rules 文件"
```

**效果**：Rules 越用越聪明，代码质量越来越高

## 5. 前后端协作

### 5.1 一人全栈（推荐）

**优势**：
```
✅ 前后端同一 Git 仓库（Monorepo）
✅ openspec/ 目录天然共享
✅ AI 同时看到前后端代码
✅ 生成的 Spec 连通性强
✅ 减少"字段对不上"的低级错误
```

**仓库结构**：
```
project-root/
├── openspec/          # 共享规范
├── backend/           # 后端代码
├── frontend/          # 前端代码
└── docs/              # 文档
```

### 5.2 双人协作

**核心原则**：**契约优先（Contract First）**

> 在双人模式下，OpenSpec 不是用来写代码的，是用来写"接口合同"的

**标准作业程序**：

```
┌─────────────────────────────────────────────────────────┐
│ 1. 后端发起 proposal                                     │
│    → 运行 /opsx:propose                                  │
│    → 生成 proposal.md                                    │
│    → 发给前端确认                                        │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ 2. 生成接口契约                                          │
│    → 运行 /opsx:ff                                       │
│    → 生成 specs/ 目录（接口定义）                        │
│    → 前后端共同 Review                                   │
│    → 确认后锁定（这就是"合同"）                          │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ 3. 同步 specs/ 到 Git                                     │
│    → 前端 git pull 查看接口定义                          │
│    → 无需等后端写完代码                                  │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ 4. 分头生成代码                                          │
│    后端：/opsx:apply → Controller/Service/Mapper        │
│    前端：/opsx:apply → API 调用/页面组件                 │
│    （双方基于同一份 specs/）                             │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ 5. 联调测试                                              │
│    → 因为基于同一份 Spec，接口天然一致                   │
│    → 减少联调问题                                        │
└─────────────────────────────────────────────────────────┘
```

**⚠️ 关键规则**：
```
❌ 严禁直接改代码！
✅ 正确做法：修改 specs/ → 运行 /opsx:apply → AI 同步代码
✅ 一方修改必须通知另一方 git pull
```

### 5.3 Git 仓库管理

**推荐**：
```
✅ 前后端同一 Git 仓库
✅ openspec/ 上传到 Git
✅ .trae/rules/ 上传到 Git
❌ skills/ 不上传（每人版本可能不同）
```

## 6. 测试规范

### 6.1 测试分类

| 类型 | 工具 | 覆盖率要求 | 说明 |
|------|------|-----------|------|
| 单元测试 | JUnit / Vitest | 核心代码 > 80% | AI 自动生成 |
| 集成测试 | SpringBootTest | 关键流程 > 90% | AI 自动生成 |
| E2E 测试 | Playwright | 核心页面 > 100% | 网页自动化 |

### 6.2 测试流程

```
1. AI 生成单元测试 → 自动执行 → 失败则自动修复
2. AI 生成集成测试 → 自动执行 → 失败则自动修复
3. Playwright 网页测试 → 输出 Excel 报告
4. 人工测试 → 最后的质量保证（不能降低要求）
```

### 6.3 人工校验点

- ✅ 验证 AI 生成的用例是否覆盖核心流程
- ✅ 对 AI 未覆盖的复杂场景人工补充
  - 弹窗处理
  - 文件上传
  - 下拉选择
- ✅ 筛选不通过、待改进用例，进行功能优化

## 7. 避坑指南

### 7.1 常见问题及解决方案

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| Maven 打包失败 | VS Code 后台索引冲突 | 关闭 `java.autobuild.enabled` |
| AI 生成魔法值 | Rules 未约束 | 完善 Rules，要求使用常量/枚举 |
| Spec 和代码不一致 | 手工修改代码 | 修改后反向同步 Spec |
| Rules 文件太长 | AI 遵循度下降 | 拆分到<200 行，或用 Skills |
| 前后端接口对不上 | 未共享 specs/ | 共享 openspec/specs/作为合同 |
| AI 忘记规则 | 规则位置不对 | 放提示词开头和结尾 |
| SpringBoot 启动失败 | 文件锁/读写冲突 | 关闭后台索引，让 Maven 独占 |

### 7.2 性能优化建议

**Rules 文件**：
```
✅ 单个文件 ≤ 200 行
✅ 按功能拆分（backend.md, frontend.md...）
✅ 复杂工作流使用 Skills（按需加载）
❌ 全部堆在一个文件里
```

**AI 会话**：
```
✅ 复杂任务分多次对话
✅ 每次对话聚焦一个主题
✅ 重要规则重复强调
❌ 一次性丢给 AI 太多内容
```
